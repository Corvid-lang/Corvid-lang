# Corvid moat benchmarks

These are not throughput benchmarks. They measure the dimensions Corvid is
designed to win on:

1. **Compile-time rejection rate** — for a fixed set of AI-application bug
   classes, how many does each stack catch *before the program runs*?
2. **Governance line-count** — for the same product implemented in Corvid,
   Python, and TypeScript, how many lines exist *only* to enforce safety,
   audit, replay, approval, or provenance?

Both publish ratios + raw numbers. Both are reproducible from a clean clone.
Both compare against representative stacks (Python + LangChain / pydantic /
mypy ; TypeScript + Vercel AI SDK / zod / strict mode) — never against
"naive Python" with no static checking.

## Why this directory exists

`benches/corvid/` measures *orchestration overhead*. Phase 17 closed with
the honest finding that Corvid is 25–36× slower than Python on those
benches. That ratio is published deliberately; raw throughput is not the
moat.

`benches/moat/` measures *what the language refuses to ship*. A program
that compiles in Python but fails Corvid's typecheck for a real bug class
is the Corvid claim, not the millisecond delta.

## Benchmark protocol

Both benchmarks follow the same shape:

1. **Equivalent implementations.** Whatever is being measured exists in
   Corvid + Python + TypeScript with the *same intended behaviour*. No
   stack has a feature the others lack at the implementation level.
2. **Representative dependencies.** Python uses `pydantic`, `mypy`, and
   the LangChain abstractions a normal developer would reach for.
   TypeScript uses `zod`, `tsc --strict`, and the Vercel AI SDK's typed
   surface. Comparing against unannotated Python or `any`-typed TS is
   forbidden — it would let Corvid win against an unfair baseline.
3. **No LLM latency.** Mock providers everywhere. Any benchmark whose
   ratio depends on real-provider latency is automatically out of
   scope; the ROADMAP rule is explicit.
4. **Deterministic runner.** Each benchmark ships a runner that takes
   the cases, runs each stack, and emits a JSON result + a Markdown
   summary. The same runner output reproduces from a clean clone.
5. **Honest classifications.** When Corvid catches a case the others
   miss, the case description names the exact compiler diagnostic
   (`approval.dangerous_call_requires_token`,
   `effect_row.body_completeness`, etc.) so reviewers can verify the
   claim against `corvid contract list`.

## How to add a new case / app

Each subdirectory's README documents the exact procedure. The principle:
adding a case must require zero edits to the runner — just drop the new
directory in and the runner picks it up. This is the "no shortcuts" rule
applied to benchmarks themselves: a benchmark is not done until it can
absorb the next case without code changes.

## How to publish numbers

Each benchmark produces a `RESULTS.md` file under its directory. The
file is regenerated by the runner; do not hand-edit. CI runs the
runner on every push (once the harness is wired) and a drift gate
fails the build if the committed `RESULTS.md` no longer matches what
the runner produces. This mirrors the `docs/core-semantics.md` drift
gate from Phase 35-D.
