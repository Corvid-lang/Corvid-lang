# Phase 42 Production Reference Applications

Phase 42 proves Corvid can build production-shaped AI backends, not prototypes.
Reference apps must run as Corvid server backends with auth, database state,
jobs, connectors, approvals, traces, evals, deployment manifests, and operator
runbooks.

## Non-Scope

Phase 42 does not ship hosted SaaS products or polished frontends. It ships
complete backend reference apps and deterministic local/mock modes that external
developers can run, inspect, replay, and deploy.

## Shared Architecture

Every reference app uses:

- Corvid server routes.
- SQLite/Postgres migrations.
- Session/API-key auth.
- Durable jobs and schedules.
- Corvid-owned connectors from Phase 41.
- Approval-gated dangerous writes.
- Lineage traces and `corvid observe` reports.
- Redacted eval promotion from trace slices.
- Mock connector mode by default.
- Real provider mode only behind documented env vars.
- Operator runbook and security model.

## App Set

1. Personal Executive Agent backend.
2. Personal Knowledge Agent backend.
3. Personal Finance Operations Agent backend.
4. Customer Support Operations Agent backend.
5. Code Review and Maintenance Agent backend.

## Quality Bar

Each app must include:

- At least 10 tables and 5 migrations.
- At least 3 connector integrations in mock mode.
- At least 5 approval contracts.
- At least 3 durable jobs.
- At least 10 eval cases.
- At least 5 named adversarial tests.
- Deployment manifest and operator runbook.
- Claim evidence generated by runnable commands.

## Security Posture

Apps must not over-claim autonomy. Any external write is approval-gated. Any
regulated or high-stakes domain states its non-goals. Token handling stays in
connector state, not app code. Trace and eval artifacts are redacted before
committing.

## Demo Mode

Demo mode must work offline:

- Seed database.
- Mock connectors.
- Deterministic jobs.
- Deterministic replay fixtures.
- No provider credentials required.

## Acceptance

Phase 42 is done when an external developer can clone the repository, run at
least one full backend reference app locally, inspect approvals/traces/evals, and
deploy it without writing a second backend in another language.
